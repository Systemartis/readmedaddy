the age of agentic development

Documentation is code now.

"It's just markdown files. Real AI work is actual, running code." — the dismissal Daniel Warfield got for a Markdown-based agent project, right before arguing the opposite ("The Documentation is the Code", 2026)

That dismissal has it backwards. In agentic development, Markdown is the part that executes: an agent reads your README to learn the quickstart, your AGENTS.md to learn the rules, your CLAUDE.md to learn the architecture, a SKILL.md to learn a whole capability — and then it writes the running code. The prose is upstream of the program.

The consequence: every stale sentence in those files is a bug an agent will faithfully reproduce, at machine speed, on every session. Docs stopped being the thing that rots quietly — the rot compiles now.
if docs are code…

…they need what code has.

We'd never ship code without version control, review, tests, and CI. Docs-as-code brought the first two. The age of agents makes the rest non-negotiable — and that's exactly the gap this project fills:

01 · The problem

A README rots the moment the code moves
and nobody notices.

The front page is built from a handful of signal files. Every one of them changes for reasons that have nothing to do with documentation — and each change quietly turns the README into a lie.

package.json · pyproject.toml · Cargo.toml

Manifests change your install line and your dependencies. The quickstart that used to work now returns an error nobody screenshots.

bin/ · src/ · cmd/ · cli/

Entrypoints change your quickstart and your API surface. The example command on the front page invokes a flag that no longer exists.

.github/workflows/ · install.sh · Makefile

CI and install scripts change your badges and your setup steps. The green badge points at a workflow that was renamed months ago.

SKILL.md

For an agent skill, this file is the product. When it changes and the README does not, the front page describes a different tool.

Nobody edits a README to break it. It breaks by omission — code moves, the page doesn't, and the first thing every visitor reads is the least reliable file in the repo.

02 · What it is

Three surfaces, one drift logic.

A self-contained folder of Markdown plus one POSIX-shell hook — no runtime dependencies, nothing to build, nothing tied to a single vendor.

Agent Skill

Plain Markdown that any capable agent can follow: Claude Code and Copilot load it natively, opencode reads the Claude path, and everything else hooks in via one AGENTS.md line. Detects the repo's archetype, drafts competing candidates, and ranks them through ten weighted gates.

Drift detector

One POSIX shell script, pure local git. Runs as a Claude Code Stop hook in-session, and standalone as --check for CI, git hooks, and any other agent. No network, no state surprises, no way to break a session.

GitHub Action

A composite action wrapping the same script with --check --range. Comments on — or fails — a pull request that leaves the README behind. Catches drift from contributors who use no agent at all.

One detector, one config. All three surfaces read the same .readmedaddy.json, so a project defines "what counts as drift" exactly once.

03 · Generation

Five steps. The fourth is the differentiator.

1
Detect
resolve the repo to one of ten archetypes from manifests & entrypoints
▶
2
Pull
load that archetype's winning patterns, sections & exemplars
▶
3
Generate
2–3 genuinely different candidates per key section
▶
4
Rank
score every candidate through 10 gates, archetype-weighted
▶
5
Assemble
stitch the winner, graft runner-up wins, verify every claim

Ten archetypes

CLI · library · framework · app/SaaS · infra/devops · data/ML · agent skill/plugin · research · monorepo · internal tool. A good README for a CLI is a bad README for a library, so each repo is scored against its own genre, never a generic checklist.

Tournament mode, when it matters

For a front page that materially moves adoption: five to seven whole-README drafts in distinct styles, scored by a three-judge panel (first-impression, craft, ASCII/design). The top draft becomes the skeleton; per-gate winners are grafted in; the merge is kept only if a re-score beats every input.

Step 5 is non-negotiable: every factual claim — install commands, file paths, version, license — is checked against the repo before a line ships. No invented benchmarks, no fake stars, no badge for a thing that does not exist.

04 · The rubric

Ten gates, weighted by archetype.

Every candidate is scored 0–5 per gate against concrete anchors. The score is the selection function, not a report card.

GateWhat it measuresGateWhat it measures
G1 Hookfirst screen: what it is + why you'd care, in one lineG6 Completenessusage, config, examples, links, license — without bloat
G2 Identity / trustname, one-liner, real badges — never a fabricated countG7 Credibilityreal examples, tests/CI signals, honest limitations
G3 Visuala wordmark, diagram, screenshot, or demo that earns its spaceG8 Contextual fitfollows the conventions of its archetype
G4 Quickstartinstall + smallest real usage in under 30s, copy-paste correctG9 Community / maintcontributing, code of conduct, changelog, support/roadmap
G5 Scannabilityheading hierarchy, short paragraphs, tables, a TOC once longG10 Voicedistinct, confident, free of AI-slop

The weighting math

Every gate starts at base weight 2. Each archetype bumps its three heaviest gates by +4, +3, +2, so every row sums to 29 and the maximum weighted score is always 29 × 5 = 145. Normalized total: round(Σ wᵢ·sᵢ / 145 × 100, 1).

A CLI bumps G3 demo (6), G4 quickstart (5), G1 hook (4). An agent skill bumps G1 hook (6), G4 examples (5), G8 fit (4).

The same numbers drive the fix list

Each gate's deficit — wᵢ × (5 − sᵢ) — is how many normalized points that gate leaves on the table. Rank fixes by deficit, highest first: a weak gate the archetype cares about always outranks a weak gate it doesn't. The rubric prioritizes the work instead of leaving every gap looking equal.

05 · The drift hook

The Stop hook notices what you didn't.

When a session's changes touch the watched signal files but not the README, the hook prompts a refresh through the skill, in the same session — while the full context is still loaded. It detects drift and asks; it never rewrites your README on its own.

ModeChannelFiresUse it when
auto defaultdecision: block on stdoutonce per distinct driftyou want one clear nudge, then silence until something new drifts
notifyone line on stderronce per distinct driftawareness without a handoff — nothing blocked, nothing continued
enforcedecision: block on stdoutevery Stop until the README changesthe project has a hard "README tracks code" rule

Loop safety, twice over

  • stop_hook_active guard — when Claude Code re-runs the hook as part of a hook-driven continuation, it exits immediately. It can never fire twice in one turn.
  • Cooldown state — a signature of the handled drift (short HEAD sha + sorted dirty watched files) is written to .git/readmedaddy-state. Identical signature next Stop → silence. enforce deliberately skips recording.

Incapable of breaking a session

  • No git repo, no README, or enabled: false → does nothing.
  • Every branch exits 0 — there is no failure path.
  • State lives inside .git/: never an untracked file in your tree, never needs a gitignore entry.
  • Session override: README_DADDY_HOOK=notify|enforce|off.
06 · Agent-agnostic --check

The same detector, anywhere. No agent required.

# working tree (+ last-commit fallback)
skills/readmedaddy/hooks/readme-drift.sh --check

# commit range, for CI
skills/readmedaddy/hooks/readme-drift.sh --check --range origin/main...HEAD
0
fresh

— or nothing to check: no git repo, no README, or enabled: false

1
drift

the drifted watched files print to stdout, one per line

2
usage / git error

bad --range — loud on purpose, so a misconfigured CI gate fails visibly

Deliberate semantics

  • Idempotent and stateless — reads no stdin, writes no cooldown state; run it twice, get the same answer twice.
  • Ignores README_DADDY_HOOK — that env var is session-scoped; an explicit --check should always answer.
  • Respects .readmedaddy.json — one config governs every surface.

Pre-commit wiring — warns, never blocks

# .git/hooks/pre-commit
~/.claude/skills/readmedaddy/hooks/readme-drift.sh --check \
  || echo "readmedaddy: consider refreshing the README before you push"

Works for every agent and every human on the machine.

07 · The GitHub Action

A PR gate for contributors who use no agent at all.

on: pull_request
permissions: { contents: read, pull-requests: write }
steps:
  - uses: actions/checkout@v4
    with: { fetch-depth: 0 }        # the range diff needs the merge-base
  - uses: Systemartis/readmedaddy@v0
    with: { mode: comment }         # one sticky PR comment; 'fail' = required check

mode: comment — advisory

  • Posts one sticky comment naming the drifted files, found by a hidden marker and updated in place on later pushes — never a pile-up.
  • On fork PRs (read-only token) it degrades gracefully: a warning plus a step summary instead of a red advisory check.
  • Drifted filenames are sanitized against markdown fence injection.

mode: fail — the gate

  • Fails the job when drift is detected — usable as a required check.
  • Makes no API calls at all in this mode.
  • Outputs for downstream steps: drift ("true"/"false") and files (newline-separated).

Pure local git — no LLM, no data leaves the runner. LLM-scored merge gates are deliberately not offered: judge scores wobble between passes, so scoring stays advisory. This repo runs the same action on its own pull requests.

08 · Local-only by design

Enforced by CI, not just promised.

No network, ever, in shipped code

The skill is Markdown; the hook, installer, and validator are POSIX shell and stdlib Python doing local git and file reads. A no-network guard in scripts/validate-skill.py fails the build if any network primitive appears in any shipped .sh or .py file:

curl · wget · urllib · socket · requests. · /dev/tcp · …

The skill orders the model offline

SKILL.md's contract: every fact comes from the local repository — no web searches, no remote templates, no URL lookups. The README canon it draws on is baked into the reference files so nothing needs fetching.

No telemetry, no accounts

Nothing is collected, counted, or reported anywhere. There is no server side. Bounded, atomic writes: one README, one cooldown file inside .git/, and (Claude Code only) a temp-file-plus-rename settings merge.

Why this matters vs SaaS drift tools

In a 28-tool survey of the drift-detection field (2026), every LLM-powered rival pays network, tokens, or a SaaS subscription for detection itself — code ships to someone's cloud just to learn whether the README is stale. readmedaddy's detection is deterministic local git: free to run on every Stop, every commit, every PR.

The one honest boundary

Your agent's own model traffic is outside readmedaddy's control. Run Claude Code or Copilot against a cloud model and your repo context goes wherever your agent sends it — for readmedaddy exactly as for every other task. Pair the skill with a locally-hosted model and the entire loop is airgap-friendly.

09 · Eval receipts

Thresholds pre-registered before any run.
The first run cleared all of them.

Hypothesis, thresholds, floor, and falsifiers were committed before execution; nothing was chosen after seeing a result. The bar was conjunctive and per-fixture: 4/4 detection, ≥ +20 lift on every fixture, an absolute floor of 70/100 — no strong average can hide a weak fixture.

FixtureArchetypeBaseline /100readmedaddy /100LiftDetection
cli-fixtureCLI15.975.2+59.3correct
lib-fixturelibrary13.171.7+58.6correct
skill-fixtureagent skill12.477.9+65.5correct
research-fixtureresearch9.789.0+79.3correct

Protocol

Mean lift +65.7 (threshold ≥ +25). Blind judging: 24 independent passes — 8 anonymized READMEs × 3 passes each; no judge saw a pair, a label, or any hint of origin. Median-of-3 per gate; weighted totals recomputed independently by a stdlib-only score.py, so judge arithmetic was never trusted.

The honest caveats — as pre-registered

The lift is measured against a deliberately thin baseline: this proves skill-vs-baseline, not every alternative approach. The same model family generated and judged (mitigated by blind isolation, not eliminated). Four of ten archetypes exercised. Per-pass judge noise ran up to ~9 points; median-of-3 is the pre-registered damping.

12 · Roadmap — v0.3.0

The guard release. shipped · v0.3.0

Approved design (2026-07-07), pre-implementation. Everything below is a commitment of direction, not a shipped feature.

Guard wizard (init)

One sitting configures everything: silent local detection pass, then at most 6 questions, each pre-answered from repo evidence. Agent-face, script-face, and an opt-in onboarding PR whose body forecasts which recent PRs would have been flagged. npm-init-style preview before any write; the default path makes zero network calls.

Merge-to-main + scheduled triggers

The action branches on the event instead of hard-skipping non-PR events: push to the default branch feeds one pinned "README health" dashboard issue (or fails the job); a weekly cron sweeps for drift that arrives without a push; merge queues (merge_group) are supported so a required check never stalls a queue.

Required-check enforcement

Via GitHub rulesets, applied through the user's own gh auth after explicit confirmation — offering dry-run evaluate before active. Classic branch protection is never replaced wholesale. Default is printing the exact recipe; never a silent skip.

Opt-in auto-fix tier

Pluggable from day one: runner off | claude | command, with the Claude path best-tested. Consent surface: an explicit @readmedaddy fix comment. The agent edits the working tree only; standardized PR mechanics do the rest. Explicitly priced, architecturally separate from detection, loop-immune by default.

Plugin packaging

Two new files make the repo a Claude Code plugin: hook and skill install/uninstall cleanly via /plugin, superseding the settings.json wiring for Claude users. install.sh remains for opencode and Copilot.

Config schema v2 + Phase 0 fixes

A guard section with one vocabulary across all surfaces, a hardened config parser, --lint-config, and a published JSON Schema. Sequenced first: fixes for hook over-nagging on fresh installs, the offline-mandate contradiction, the README's stale action snippet (plus a moving @v0 tag), and config misparse never resolving to blocking.

13 · Philosophy

Detection is free and deterministic.
The LLM only ever writes prose
a human reviews.

  • Pure local git decides whether the README is stale — no tokens, no network, the same answer twice. That's why it can run on every Stop, every commit, and every PR without costing anything.
  • The model only drafts prose — ranked through ten gates, verified against the repo, and reviewed by you. The hook never edits your files; the action never contacts a third party.
  • Every quality claim traces to a rubric score or a pre-registered eval. The score is earned, not asserted — and the eval can say no.
